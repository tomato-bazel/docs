## Known issue: PanelDescriptor JSON ↔ wasm serde oneof mismatch

**Status:** open — to fix in a future session. The web renderer's wasm core cannot
deserialize the `PanelDescriptor` JSON shape that the TS renderer and the demo
produce, so `examples/uiview-demo` does not actually render a table.

### Symptom

Rendering any `TablePanel` through the web renderer fails with
**`descriptor body is not TABLE`**, thrown from `rust/uiview/src/wasm_api.rs`
(`render_table_wasm` and `build_populate_request`, where they match
`d.body == Some(Body::Table(..))`).

### Root cause

The `body` oneof of `PanelDescriptor` is represented two **incompatible** ways:

- **TS / demo / docs use proto3-JSON flat** — `src/uiview/renderer.ts` routes on
  `descriptor.table`; `examples/uiview-demo/demo.js` hand-builds
  `{ panel_id, title, table: { … } }`; `wasm_api.rs` itself documents "snake_case
  keys, matching proto3 JSON." Proto3-JSON puts the active oneof variant as a
  top-level field: `{ "table": { … } }`.
- **The wasm uses serde-default externally-tagged** — `wasm_api.rs` does
  `serde_wasm_bindgen::from_value::<PanelDescriptor>(…)`, and prost's serde derive
  produces `{ "body": { "Table": { … } } }` (a `body` object whose key is the
  PascalCase variant name).

A shape probe (feeding each candidate into the prebuilt wasm and calling
`renderTable`) confirmed: **only `{ body: { Table: … } }` is accepted**;
`{ table: … }` and `{ body: { table: … } }` both fail. So the renderer and its own
core disagree on the descriptor shape, and the demo never rendered.

### Why the obvious fix does NOT work

Adding `#[serde(flatten)]` to the `body` field and
`#[serde(rename_all = "snake_case")]` to the `Body` oneof (via prost-build
`field_attribute` / `type_attribute`) and rebuilding the wasm
(`wasm-pack build rust/uiview --target web`, which auto-installs the matching
wasm-bindgen `0.2.105`) **compiles, and the attributes land correctly in the
generated code — but it still fails.** Reason: **serde's `#[serde(flatten)]` is
unsupported on externally-tagged enums**, so flatten cannot produce the
proto3-JSON oneof representation. This was tried and reverted; don't repeat it.

### The fix: pbjson

Generate **proto3-JSON-compatible serde** with **pbjson** (`pbjson` +
`pbjson-build`), which represents oneofs as flattened field names correctly. It
must be applied to **both** codegen paths, because the canonical wasm is
bazel-built:

1. **Cargo** — `rust/uiview/build.rs`: have prost-build emit a file-descriptor set
   (`Config::file_descriptor_set_path(…)` + `compile_protos(…)`), then run
   `pbjson_build::Builder` over it to generate the serde impls and `include!` them
   next to the prost types. Add `pbjson` (dep) and `pbjson-build` (build-dep).
2. **Bazel** — `//rust/uiview:uiview_wasm` builds via the `bazel_proto` path
   (`uiview_proto`, generated by the `prost_toolchain` in `rust/BUILD.bazel`). The
   same pbjson serde must be produced there — configure the proto codegen to emit
   pbjson impls, or generate them in a small wrapper crate the wasm lib depends on.

**Caveat:** proto3-JSON serializes enum fields (`ColumnFormat`, `ContextSource`)
as enum **name strings**, while `demo.js` currently uses integers (e.g.
`format: 2`). pbjson accepts both, but verify the column `format` values still
parse after the switch.

### Verify

Rebuild the wasm, then load a flat descriptor
(`{ panel_id, title, table: { populate, rows_field, columns: [...] } }`) through
`renderPanel`. It should render a table, and `examples/uiview-demo` should render
end to end.

### How this surfaced

Found while embedding meridian's web renderer in a Servo-based host window
(`miniservo`), with the `RpcInvoker` served over a native `rpc:` URL scheme. The
host and the transport worked, and meridian's **core** (`renderTable` — the
proto-walk + `ColumnFormat` formatting) rendered the table when fed the
`{ body: { Table: … } }` shape directly; only `renderPanel` (the TS DOM glue) hit
this mismatch. So the core path is a viable interim renderer until the serde fix
lands. (Full context: the miniservo↔meridian backend spike in the citizen-sh repo.)
